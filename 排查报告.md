# 服务无法整体启动排查报告

## 1. 排查目标
- 目标机器：`192.144.227.251`
- 登录账号：`root`
- 项目目录：`/root/code/bus_gallery/`
- 排查问题：前端可访问但接口大量 `502`，系统“整体跑不起来”

## 2. 排查思路
按以下顺序排查，避免只看单点日志误判：
1. 先确认 `docker-compose` 实际目录与当前在线容器集合。
2. 再看关键链路容器状态（`gateway -> bridge/group -> nacos/mysql`）。
3. 复现场景并抓最新报错（区分历史日志和当前故障）。
4. 校验宿主机资源是否满足多 Java 服务并发启动。
5. 输出根因分层和修复优先级。

## 3. 关键操作与结果

### 3.1 compose 目录确认
- 在 `/root/code/bus_gallery/` 直接执行 `docker compose ps` 报错：无配置文件。
- 发现实际 compose 文件在：`/root/code/bus_gallery/docker/docker-compose.yml`。

### 3.2 当前容器状态
在 `/root/code/bus_gallery/docker` 下查看状态：
- 运行中：`frontend/backend/mysql/redis/rabbitmq/minio`
- 已退出：`gateway/bridge/group/nacos`

`docker ps -a` 显示：
- `bus-gallery-nacos`：`Exited (128)`，且退出已持续较长时间。
- `bus-gallery-gateway/bridge/group`：均未处于可服务状态。

### 3.3 前端到网关链路验证
- 在 `frontend` 容器内访问 `http://gateway:8094/actuator/health`：
  - 返回：`wget: bad address 'gateway:8094'`
- 说明 `gateway` 容器未加入当前可解析服务集合（实质是未运行），前端出现 502 属必然现象。

### 3.4 复现启动失败
执行：
- `docker compose up -d nacos group bridge gateway`

关键报错：
- `failed to create task for container ...: already exists`
- `docker inspect bus-gallery-nacos` 中 `State.Error` 同样为上述错误。

说明：
- `nacos` 容器卡在容器运行时层（task 残留/冲突），不是单纯应用代码报错。

### 3.5 日志证据（应用层）
`gateway/group` 历史日志出现：
- `UnknownHostException: nacos`
- `UnknownHostException: mysql`（group 启动阶段）
- `NacosException: Client not connected`

这与 `nacos` 长期不可用相互印证：服务注册/发现链路断裂，`gateway/bridge/group` 无法完成启动。

### 3.6 资源证据（宿主机）
宿主机资源：
- 内存总量约 `2.0G`，`Swap=0`
- 各 Java 服务 Dockerfile 默认：`-Xms256m -Xmx512m`
- Nacos 镜像默认 JVM 参数含 `-Xms1g -Xmx1g -Xmn512m`

日志里已有直接证据：
- `There is insufficient memory for the Java Runtime Environment to continue`
- `Native memory allocation (mmap) failed`

说明：
- 即使处理了 `nacos task already exists`，当前机器 2G 无 swap 的预算也很难稳定承载全套服务。

## 4. 根因结论（按优先级）

### P0：Nacos 容器处于运行时异常
- 表现：`Exited(128)` + `task already exists`，无法正常拉起。
- 影响：`gateway/bridge/group` 依赖 Nacos，服务发现链路中断，API 网关无法工作，前端统一表现 502。

### P0：宿主机内存预算不足
- 表现：多服务并发启动出现 JVM 内存申请失败。
- 影响：即便个别服务短时拉起，也会反复崩溃，系统不可稳定运行。

### P1：链路启动顺序受上游不可用放大
- `group/bridge/gateway` 强依赖 `nacos` 与 `mysql` 解析可用性，`nacos` 不可用后触发级联失败。

## 5. 建议修复顺序（可执行）

1. 先修复容器运行时异常（Nacos）
- 删除异常 `nacos` 容器并重新创建，必要时重启 docker daemon 清理残留 task。
- 确认 `nacos` 成功 `Up` 且 `8848/9848` 可用后再启动下游。

2. 再处理内存容量问题（必须）
- 推荐至少升配到 `4G`（更稳妥 `8G`）或增加 swap。
- 同时下调 Java 堆参数（尤其 Nacos 和 gateway/group/bridge）以匹配机器规格。

3. 按依赖顺序启动并验活
- `mysql/redis/rabbitmq/minio -> nacos -> backend/group -> bridge -> gateway -> frontend`
- 每一步执行健康检查，失败即停并看最近 100 行日志。

4. 通过后再回归前端接口
- 在 `frontend` 容器内验证 `http://gateway:8094/actuator/health` 可达，再验证登录与业务接口。

## 6. 一句话总结
这次故障不是单模块代码问题，而是“**Nacos 容器运行时异常 + 2G 无 swap 内存不足**”导致的基础设施级联失败；网关未存活，前端出现 502 是结果而不是原因。

## 7. 实际修改过程（JVM 调优 + Bug 修复）

### 7.1 JVM 与资源调优（已执行）
1. 增加并持久化 swap：
- 新建 `/swapfile`（2G），`swapon` 启用，并写入 `/etc/fstab`。
- 目的：缓解 2G 内存下多 Java 进程同时启动时的 OOM/Native memory 失败。

2. 新增低内存覆写文件：
- 文件：`docker/docker-compose.lowmem.yml`
- 关键调整：
  - `backend`：`-Xms128m -Xmx320m`
  - `group`：`-Xms128m -Xmx256m`
  - `bridge`：`-Xms96m -Xmx224m`
  - `gateway`：`-Xms128m -Xmx256m`
  - 降低连接池规模（`backend/group` 的 Hikari pool）与日志级别，减少内存与 IO 压力。

### 7.2 Nacos 相关修复尝试（已验证）
1. 先后处理了 Nacos 的多个启动阻塞点：
- 容器 task 残留（`task already exists`）；
- 启动参数与鉴权变量缺失；
- JWT token 长度不满足要求；
- 后续又遇到 Derby 初始化超时（`load derby-schema.sql error`）。

2. 结论：
- 在当前 2G 机器上，本机 Nacos 作为强依赖不稳定，继续强绑会反复放大为网关 502。

### 7.3 502 核心 Bug 修复（代码改动）
为绕开 2G 场景下 Nacos 强依赖，改为“低内存 + 直连”运行模式（仍保留回滚能力）：

1. Gateway 路由支持环境变量直连：
- 修改：`gateway/src/main/resources/application.yml`
- 将 `lb://...` 路由改为可配置：
  - `GATEWAY_BRIDGE_URI`（默认仍可回到 `lb://bus-gallery-bridge`）
  - `GATEWAY_CONTENT_URI`（默认仍可回到 `lb://bus-gallery`）

2. Bridge 的 Feign 客户端支持 URL 直连：
- 修改：`bridge/src/main/java/com/busgallery/bridge/client/BridgeContentClient.java`
- 修改：`bridge/src/main/java/com/busgallery/bridge/client/BridgeTradeClient.java`
- 为 `@FeignClient` 增加 `url` 配置，支持不经服务发现直接访问 `backend/group`。

3. Bridge 配置新增直连项：
- 修改：`bridge/src/main/resources/application.yml`
- 新增：
  - `bridge.content-service-url`
  - `bridge.trade-service-url`

4. 低内存覆写中启用直连运行参数：
- `NACOS_DISCOVERY_ENABLED=false`
- `BRIDGE_CONTENT_SERVICE_URL=http://backend:8080`
- `BRIDGE_TRADE_SERVICE_URL=http://group:8092`
- `GATEWAY_BRIDGE_URI=http://bridge:8093`
- `GATEWAY_CONTENT_URI=http://backend:8080`

### 7.4 运行期数据与环境 Bug 修复（已执行）
1. 修复 `group` 反复重启：
- 根因：`Unknown database 'trade_center'`
- 处理：创建 `trade_center` 并导入 `trade_center.sql`。

2. 修复前端仍 502（上游地址缓存问题）：
- 现象：前端 Nginx 仍指向旧网关容器 IP（日志有 `Host is unreachable`）。
- 处理：重启 `frontend` 容器刷新上游 DNS 解析。

### 7.5 最终结果（本次实操验收）
1. 容器稳定性：
- `backend/group/bridge/gateway` 最终均为 `running`，`RestartCount=0`（验收时）。

2. 接口结果：
- `GET /api/vehicles?size=1`：`200`（内容链路恢复）。
- `GET /api/bridge/portal/teams?...`：`401`（未携带登录态，属于鉴权预期，不是 502）。
- `POST /api/auth/login`：返回业务校验错误（`400`），说明已到应用层，不是网关不可达。

3. 结论：
- 已从“网关不可达导致全站 502”恢复到“业务可达 + 鉴权/参数按预期返回”状态。
